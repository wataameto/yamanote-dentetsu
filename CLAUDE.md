# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

еұұжүӢз·ҡйӣ»йү„ (Yamanote-Dentetsu) вҖ” a Momotetsu (жЎғеӨӘйғҺйӣ»йү„) -style board game built with Phaser 3 and Vite. No backend; everything runs client-side. Players race around a hexagon-shaped Yamanote-line loop (with a Chuo-line shortcut branch) buying properties, drawing cards, and dodging гғҺгғ©гғҚгӮі (a lightweight "poverty cat" mechanic), aiming for the highest total assets after N years.

See `DESIGN.md` for the original game-design research (Momotetsu mechanics) and phase plan; this file documents the actual as-built architecture.

## Commands

```
npm run dev      # start Vite dev server (http://localhost:5173)
npm run build     # production build to dist/
npm run preview   # preview the production build
```

No tests, no linter configured. When testing interactively with Playwright, run the dev server on a port other than 5173 (so as not to collide with the user's own session) and temporarily add `window.__game = new Phaser.Game(config)` in `main.js` for introspection вҖ” always revert that line before committing.

## Scenes

- **`src/scenes/TitleScene.js`** вҖ” title screen. Year-count picker (3Г—3 grid: 1/3/10/30/100/300/1000/3000/10000 years вҖ” yes, the joke long modes are real options) and a "гҒӨгҒҘгҒҚгҒӢгӮү" (continue) button that opens a 10-slot load modal (see Save below). Shares a single `SFX` instance and BGM playback with `GameBoardScene` via `this.registry` (`registry.get('sfx')`), so the theme keeps playing seamlessly across the scene transition.
- **`src/scenes/GameBoardScene.js`** вҖ” the entire game. Large single-scene class (~1600 lines) owning board state, HUD, turn flow, cards, properties, and modals. `create(data)` branches: fresh game (`setupNewGame`) vs. loaded save (`loadFromSaveData`, from `data.loadData`).

## Board model (`src/board.js`, `src/layout.js`, `src/stations.js`)

- `STATIONS` (30 Yamanote stations, `stations.js`) are laid out by `buildStationPositions()` (`layout.js`) into a hexagon shape with diagonal "shoulder" sections at еӨ§еҙҺ/й«ҳијӘгӮІгғјгғҲгӮҰгӮ§гӮӨ (bottom) and е·ЈйҙЁ/з”°з«Ҝ (top) вҖ” extracted from an earlier tower-defense project's station-select screen.
- `buildBoard()` builds `mainLoop`: 60 cells, alternating `station` and small event cells (`blue`/`red`/`card`, weighted random via `pickSmallType`). Station `i` always occupies main-loop cell `2*i`.
- A separate `chuoPath` (5 cells: small, еӣӣгғ„и°·, small, еҫЎиҢ¶гғҺж°ҙ, small) models the Chuo-line shortcut connecting ж–°е®ҝвҮ„зҘһз”°.
- **Position representation**: `pos = { onChuo: boolean, index: number, chuoDir?: 1 | -1 }`. `chuoDir` only matters while `onChuo` is true вҖ” it says which direction the piece is currently walking along the 5-cell chuo path (`+1` = toward зҘһз”°, `-1` = toward ж–°е®ҝ). It is **not** meant to persist meaningfully across turns; every time movement is computed from an on-chuo position, both directions are freshly recomputed (see Movement below) and `chuoDir` on the actual player position gets overwritten in `beginMove` right before animating, so a stale value from a previous turn never leaks in.
- `stepForward`/`stepBackward` step the main loop in `+1`/`-1` cell direction. Both now support entering the chuo branch from **either** junction (ж–°е®ҝ or зҘһз”°) regardless of which of the two you call вҖ” the direction you were already traveling in on the main loop no longer restricts which junction you can dive in from. Once `onChuo`, both delegate to the shared `stepChuo()`, which walks by `pos.chuoDir` and exits to whichever main-loop junction is on that side. This means `stepForward`/`stepBackward` behave identically once on chuo вҖ” the distinction only matters for the main-loop-only portion of a path.
- `getCell(board, pos)` reads the cell at a position; `mainLoopDistance` is a rough forward/backward distance helper used for the гғҺгғ©гғҚгӮі "farthest player" calculation (see below) and CPU target-seeking heuristics.

## Movement / direction-choice system (`GameBoardScene.js`)

Momotetsu-style dice movement was reworked several times during development; the current model:

1. `rollAndChooseMove(player, diceCount)` rolls `diceCount` d6, then calls `computeDirectionOptions(player, totalSteps)`.
2. **If not on chuo**: up to 2 base options вҖ” `еҸҚжҷӮиЁҲеӣһгӮҠ`(ccw, `stepForward`) and `жҷӮиЁҲеӣһгӮҠ`(cw, `stepBackward`), always using the *full* roll (no "stop early" вҖ” that was tried and removed; see git history). Additionally, for **each** of the 4 combinations of (forward/backward) Г— (via ж–°е®ҝ/via зҘһз”°), `findChuoBranchPassThrough` checks whether that path passes through the junction cell mid-route (or the player is already sitting on it вҖ” handled by a dedicated `chuoEntryHereOption` special case, to avoid the direction check firing twice for the "already at the junction" case and creating a duplicate option). If it does, and steps remain, a рҹҡғ chuo option is added.
3. **If already on chuo**: `computeChuoDirectionOptions` computes the two possible travel directions (toward ж–°е®ҝ / toward зҘһз”°) fresh, ignoring any previously-stored `chuoDir`. Each direction:
   - If the roll doesn't reach that exit: single option, landing inside the chuo path.
   - If it lands exactly on the exit: single option, landing on that station.
   - If it overshoots the exit: **forks** into continuing ccw or cw on the main loop for the leftover steps вҖ” two options.
   - So a single roll from mid-chuo can yield up to 4 options total (2 directions Г— up to 2 forks each).
4. **Arrow-key assignment**: because the board is a loop, "ccw" doesn't map to a fixed screen direction вҖ” on the left straight column ccw is "down", on the right column it's "up", and at й§’иҫј/е“Ғе·қ it's left/right. Every option's key is computed from real geometry: `arrowKeyForStep(fromPos, oneStepTowardOption)` diffs pixel coordinates and returns whichever of Up/Down/Left/Right has the larger component. Collisions fall back to whatever arrow key isn't taken yet (`pickKey`/inline fallback logic). Do **not** reintroduce a fixed "near exit вҶ’ Up/Down, far exit вҶ’ Left/Right" scheme вҖ” that was tried and shipped a real bug (fourи°·вҶ’еҫЎиҢ¶гғҺж°ҙ, which is visually to the right, got assigned `вҶҗ`) because it ignored actual geometry.
5. **Display order is fixed, independent of key geometry**: `showMoveChoiceModal` sorts the computed options into a constant `вҶ‘вҶҗвҶ’вҶ“` order (`KEY_ORDER`) before rendering rows, rather than showing them in whatever order `computeDirectionOptions` produced вҖ” otherwise the up/down rows would visually swap depending on which station you rolled from, which read as "broken" even though the underlying key assignment was correct.
6. `beginMove` logs the choice, plays the shortcut jingle if `shortcutAtStep` is set, force-sets `player.pos.chuoDir` to the chosen branch's direction if starting from chuo (overriding stale state), then runs `animateSteps` (recursive, one cell per `this.delay(...)` tick вҖ” see Speed below for why the per-step tick uses a different multiplier than everything else).

CPU movement (`cpuChooseOption`) picks the option landing exactly on the target if one exists, otherwise the option minimizing `approxDistanceToTarget` (a distance estimate that also handles the chuo branch by checking both exits).

## Turn flow highlights

- `checkTargetAtTurnStart` вҖ” target arrival is awarded not just by landing on it mid-move, but also if a player is *already standing* on the target station when their turn starts (e.g., after being sent there, or the target rotates onto them). This fixed a bug where е“Ғе·қ (the shared start station) could never be claimed as a target because everyone started there and no one ever "arrived".
- `awardTargetArrival` gives the cash bonus, plays `celebrateGoal()` (3 staggered particle waves + a bounce-in bonus-amount text + a camera shake, deliberately drawn out to feel like an event), and reassigns гғҺгғ©гғҚгӮі to whichever player is farthest from the *old* target вҖ” using plain `(cellIdx - targetCellIdx + len) % len` (not `mainLoopDistance`, which has a `|| len` fallback that would wrongly mark the just-arrived player, sitting exactly on the target, as "farthest").
- Starting positions for the 4 players (you + CPU1-3) are evenly spaced around the loop (quarter-loop offsets) but the rotation offset is randomized per game (`Math.random()` in `setupNewGame`) вҖ” they used to be fixed to the same 4 stations every game, which was reported as a bug ("random spawn is always the same").
- **Yearly settlement (`runSettlement`) is a blocking modal, not a log line.** `runSettlement()` only computes/applies each player's income and returns `{player, total}[]`; `showSettlementModal(year, results, onClose)` then lists every player's result and won't call `onClose` (which resumes `continueEndTurn`/the next player's turn, or `endGame` on the final year) until the human clicks OK. This replaced an earlier version that just called `this.log(...)` once per player in a loop вҖ” since `log()` overwrites a single Text object with no delay between calls, only the *last* player's line was ever actually visible on screen.
- **Log-overwrite is a recurring footgun вҖ” watch for it whenever adding a new `this.log()` call right after another one.** `this.log(msg)` just does `logText.setText(...)`; it has no history/queue. Several bugs of this shape have already been fixed: the settlement loop above, a CPU buying multiple properties at one station (now batched into a single combined log line inside `arriveAtStation`'s CPU-purchase branch instead of one line per property), a CPU using an attack card and then immediately rolling dice in the same tick (now `this.delay(1.5, ...)` is inserted in `cpuTakeTurn` before the roll so the card-use log survives), and гғҺгғ©гғҚгӮі's transfer/reassignment messages stomping the cell-resolution or goal message that just printed (now both are wrapped in a `this.delay(...)` in `afterCellResolved`/`awardTargetArrival`). If you add a new log call that fires synchronously after another one, either merge them into one message or delay one of them.
- **Station buttons are clickable for info, not just decoration.** Each main-loop station's `bg` has a `pointerdown` calling `logStationProperties(stationIndex)`, which logs that station's full property list (name/price/owner-or-з©әгҒҚ) in one line. Chuo stations (еӣӣгғ„и°·/еҫЎиҢ¶гғҺж°ҙ) have no properties and no such handler.

## Cards / Properties / Save

- `src/cards.js` вҖ” 12 cards across 5 categories (progress/move/attack/money/defense), weighted random draw.
- `src/properties.js` вҖ” 2-4 properties per station scaled by `rank` (`propertyCountForRank`). Each property gets a flavor name from the per-station `PROPERTY_NAMES` table (indexed the same order as `STATIONS`) вҖ” picked to evoke a real landmark/specialty of that station, deliberately avoiding anything adult-flavored (a few early drafts referencing love hotels, maid cafГ©s, etc. were caught and replaced; if adding new names, keep them PG). Price is `basePrice(rank) Г— tierMultiplier Г— jitter`, where `tierMultiplier` comes from `TIER_MULTIPLIER_RANGES[count]` вҖ” a per-slot range (e.g. the 4-property tier is roughly `[1, 1] / [2.5, 4.5] / [7, 14] / [18, 38]`) so that within one station the cheapest and priciest properties differ by anywhere from a few times to several dozen times, not just a flat per-slot increment. `buildProperties` sorts each station's property array **descending by price** before returning it, so both the purchase modal and `logStationProperties` naturally list the priciest property first.
- `src/save.js` вҖ” 10 localStorage slots (`yamanote_dentetsu_save_${slot}`, key via `slotKey`). Save data is the whole scene state as plain JSON (board, properties, players, target, month/year, etc. вҖ” no functions, so it round-trips directly). `saveGame`/`loadGame`/`hasSave`/`slotSummary` are the core read/write API; `downloadSave(slot)` and `uploadSaveToSlot(slot, onDone)` were added on top for portability вҖ” `downloadSave` just wraps the existing localStorage JSON in a `Blob`+`<a download>` click, `uploadSaveToSlot` creates a hidden `<input type=file>`, reads it with `FileReader`, and writes the parsed JSON straight into that slot's localStorage key (working on an *empty* slot is fine вҖ” it's how you'd restore a save onto a fresh browser/profile). Both the in-game save modal (`openSaveModal`) and the title screen's load modal (`openLoadModal`) render a в¬Ү/в¬Ҷ icon button pair next to every slot's row, not just the main save/load button.

## Speed setting

Two independent knobs, both driven by `this.stepMs` (not the old single `speedFactor` design вҖ” see below):

- `this.speedLevel` (1-10, default 5, persisted in `this.registry`) indexes `STEP_SECONDS_TABLE`, a literal seconds-per-tick value per level (e.g. level 5 вҶ’ `0.55`), assigned to `this.stepMs = STEP_SECONDS_TABLE[level-1] * 1000`. **Lower number = faster** (shorter delay), per explicit user preference вҖ” this is the opposite of what you'd naively guess from a "speed" label, so don't "fix" it back to an inverse relationship. Selected via `openSpeedModal` (top-right "гҒҜгӮ„гҒ•" button), which shows all 10 levels in a grid with the actual seconds-per-tick printed under each number.
- `this.delay(ticks, cb)` is `this.time.delayedCall(ticks * this.stepMs, cb)` вҖ” every wait in the scene (CPU "thinking" pause, post-move cell-resolve pause, end-of-turn pause, гғҺгғ©гғҚгӮі log delays, etc.) is expressed as a tick-count multiple of this shared base, e.g. `this.delay(1.5, () => this.endTurn())`, `this.delay(3, () => this.cpuTakeTurn())`.
- **Exception**: the per-cell token-movement animation inside `animateSteps` uses a separate constant `MOVE_STEP_TICKS` (currently `0.4`) instead of `1`, so that raising "гҒҜгӮ„гҒ•" affects movement pacing at a different (faster) rate than every other wait вҖ” this was an explicit request ("movement only faster, other waits unchanged"), not a general slowdown/speedup of `STEP_SECONDS_TABLE` itself. If a future ask is "make X faster but not Y", look for whether it should be a new dedicated `*_TICKS` constant like this one rather than touching `STEP_SECONDS_TABLE`.
- When changing global pacing (not just movement), edit `STEP_SECONDS_TABLE`'s values directly вҖ” that's the one place that has needed several rounds of "still too fast, slow down more" tuning.

## HUD layout gotchas

The board fills nearly the whole screen, so HUD elements are wedged into whatever empty space the hexagon layout leaves, computed from `this.layout` (`points`, `chuoPoints`, `buttonWidth/Height`, `centerX`) rather than hardcoded pixels:

- Player list, then the dice button directly below it, both live in the gap above еӣӣгғ„и°·/еҫЎиҢ¶гғҺж°ҙ (`topBandBottom` to `chuoTop`), nudged up from dead-center by a small `UP_SHIFT` constant per explicit request ("a bit higher"). The dice button is intentionally styled *unlike* a station button вҖ” a gold pill shape (`radius: diceH/2`, distinct fill/stroke colors) with a continuously pulsing text tween вҖ” specifically so it doesn't read as "just another station box"; don't reuse `BUTTON_FILL`/`BUTTON_STROKE` defaults for it.
- Hand-of-cards UI (`renderHand`) anchors off `this.chuoBottom` (just below еӣӣгғ„и°·/еҫЎиҢ¶гғҺж°ҙ), not off the dice button's position вҖ” these two used to be coupled (hand position derived from `diceY`), which broke when the dice button moved; if you relocate one of these elements, check whether the other secretly depended on its old position.
- Message log (`logText`) sits at the height of дә”еҸҚз”°/з”°з”ә (one row up from еӨ§еҙҺ/й«ҳијӘгӮІгғјгғҲгӮҰгӮ§гӮӨ), horizontally centered in the open gap between the two side columns вҖ” not below е“Ғе·қ (an earlier position) and not literally overlapping the еӨ§еҙҺ/й«ҳијӘгӮІгғјгғҲгӮҰгӮ§гӮӨ row (there's only ~6px of clearance there, not enough for a text line). It uses `wordWrap: { width: 380, useAdvancedWrap: true }` and `origin(0.5, 1)` so long messages wrap to multiple lines and grow *upward* from a fixed bottom anchor rather than overflowing sideways into the station columns. `useAdvancedWrap` is required вҖ” Phaser's default word-wrap only breaks at space characters, and Japanese text has none, so without it a long message silently renders as one un-wrapped, way-too-wide line.
- Goal(зӣ®зҡ„ең°) station highlight and the dice button are deliberately different hues (coral-orange `0xff7043` vs. sky blue `0x4fc3f7`) вҖ” they used to both be near-identical gold tones, which read as "the same thing" at a glance. If adding new highlighted UI elements, sanity-check the palette against both of these.
- The рҹҺҜ target marker badge sits at the station box's top-right corner, nudged slightly *into* the box (`-16`/`+2` offsets, not `-6`/`-6`) rather than poking out past the edge.
- Player tokens are offset left of each station's button center (`tokenBaseOffsetX`) so they don't cover the station name; your own token (рҹҗ¶ вҖ” a dog *face* emoji, chosen over the plain side-view рҹҗ•) additionally gets a pulsing gold ring (`createToken`) so it's distinguishable from the 3 CPU tokens at a glance.
- Small event-cell squares (blue/red/card) sit outside the station button on the straight left/right columns (`outwardOffsetX`, compares cell x to `centerX`), but the two cells that are literally the ж–°е®ҝвҮ„еӣӣгғ„и°· and еҫЎиҢ¶гғҺж°ҙвҮ„зҘһз”° seams are left unshifted so they sit exactly on the seam.
